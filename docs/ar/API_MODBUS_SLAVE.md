# دليل استخدام واجهة برمجة تطبيقات Modbus التابعة (Slave)

يصف هذا المستند كيفية استخدام واجهة برمجة تطبيقات Modbus التابعة (Slave) من Aoba في تطبيقات Rust لعرض البيانات على أجهزة Modbus الرئيسية. تشمل حالات الاستخدام النموذجية خطوط الإنتاج الصناعية وأنظمة التحكم في العمليات ومنصات الاختبار.

المثال المرجعي هو حزمة `examples/api_slave`.

## 1. نظرة عامة

توفّر Aoba واجهة برمجة تطبيقات من جانب الجهاز التابع تعكس نمط واجهة الجهاز الرئيسي، مبنية على نمط Builder + Hook. وهي مفيدة عندما ترغب في:

- تحويل عمليتك إلى جهاز Modbus تابع، يعرض بيانات الملفات/السجلات للأجهزة الرئيسية الخارجية؛
- بناء جهاز Modbus قابل للتكوين بسرعة لاختبارات التكامل أو المحاكاة؛
- إرفاق سلسلة وسيطة من hooks للتسجيل والإحصائيات والتحكم في الوصول والتنبيهات.

لا تزال نقطة الدخول الرئيسية هي `_main::api::modbus::ModbusBuilder`، لكنك تستخدم `new_slave` / `build_slave`:

```rust
use _main::api::modbus::{ModbusBuilder, ModbusHook, ModbusResponse, RegisterMode};
```

## 2. دورة الحياة الأساسية للجهاز التابع

تبدو نسخة مبسطة من الجهاز التابع النموذجي كالتالي:

```rust
use anyhow::Result;
use std::sync::Arc;
use _main::api::modbus::{ModbusBuilder, ModbusHook, ModbusResponse, RegisterMode};

struct ResponseLoggingHook;

impl ModbusHook for ResponseLoggingHook {
    fn on_before_request(&self, _port: &str) -> Result<()> {
        Ok(())
    }

    fn on_after_response(&self, port: &str, response: &ModbusResponse) -> Result<()> {
        log::info!(
            "sent response on {}: station={}, addr=0x{:04X}, values={:04X?}",
            port,
            response.station_id,
            response.register_address,
            response.values
        );
        Ok(())
    }

    fn on_error(&self, _port: &str, error: &anyhow::Error) {
        log::warn!("error: {}", error);
    }
}

#[tokio::main]
async fn main() -> Result<()> {
    env_logger::builder()
        .filter_level(log::LevelFilter::Info)
        .init();

    let args: Vec<String> = std::env::args().collect();
    let port = if args.len() > 1 { &args[1] } else { "/tmp/vcom2" };

    let hook: Arc<dyn ModbusHook> = Arc::new(ResponseLoggingHook);

    let _slave = ModbusBuilder::new_slave(1)
        .with_port(port)
        .with_register(RegisterMode::Holding, 0, 5)
        .with_timeout(1000)
        .add_hook(hook)
        .build_slave()?;

    // إبقاء الجهاز التابع قيد التشغيل والاستماع لطلبات الجهاز الرئيسي
    tokio::signal::ctrl_c().await?;
    Ok(())
}
```

### معاملات التكوين الأساسية

- **المنفذ**: نفس التنسيق المستخدم للجهاز الرئيسي (`/dev/ttyUSB*`، `/dev/ttyS*`، `/tmp/vcom2`، إلخ)؛
- **معرف المحطة**: يجب أن يطابق معرف المحطة الذي ستستخدمه الأجهزة الرئيسية عند التحدث إلى هذا الجهاز التابع؛
- **وضع السجل ونطاق العناوين**: يحدد أي جزء من فضاء عناوين Modbus يعرضه هذا الجهاز التابع؛
- **المهلة**: تُستخدم داخليًا للتحكم في مهل IO/المعالجة (عادةً ما تكون متوافقة مع إعدادات الجهاز الرئيسي).

## 3. سلسلة hooks الوسيطة

على جانب الجهاز التابع يمكنك أيضًا تسجيل عدة hooks لتشكيل سلسلة وسيطة. المسؤوليات النموذجية:

- التحقق من صحة أو فحص الطلبات الواردة قبل معالجتها؛
- تسجيل ومعالجة الاستجابات بعد إرسالها؛
- إطلاق تنبيهات أو تحديث الإحصائيات عند حدوث أخطاء.

تعرض حزمة `examples/api_slave` ثلاثة hooks متسلسلة:

- `RequestMonitorHook`: يراقب الطلبات ويسجل/ينبه عند الأخطاء؛
- `ResponseLoggingHook`: يسجل كل استجابة مع عنوان السجل والقيم؛
- `StatisticsHook`: يتتبع أعداد الطلبات.

يتيح لك هذا النمط إبقاء الاهتمامات المتقاطعة (التسجيل، المقاييس، التحكم في الوصول، تحديد المعدل، إلخ) خارج منطق عملك الأساسي وإرفاقها تصريحيًا بمثيل الجهاز التابع.

## 4. حالات الاستخدام النموذجية

تشمل حالات الاستخدام الشائعة لواجهة برمجة التطبيقات التابعة في البيئات الصناعية وإعدادات الاختبار:

1. **محاكي أجهزة قائم على البرمجيات**
   - عندما لا تكون الأجهزة الحقيقية متاحة بعد، قم بمحاكاة جهاز Modbus في Rust؛
   - قم بتحديث قيم السجلات الداخلية دوريًا وفقًا لسيناريوهات الاختبار الخاصة بك؛
   - قم بتشغيل اختبارات التكامل الشاملة في CI.
2. **طبقة تكييف البروتوكول**
   - قد تتحدث أجهزتك الفعلية بلغة CAN أو TCP خاص أو ناقل ميداني آخر، بينما تتوقع الأنظمة الأعلى Modbus؛
   - استخدم واجهة برمجة التطبيقات التابعة لتعيين تلك الإشارات إلى فضاء سجلات/ملفات Modbus وتقديم واجهة Modbus موحدة.
3. **بوابة طرفية تعرض بيانات معالجة**
   - جمع وتطبيع البيانات من مصادر متعددة داخل عمليتك أو بوابتك؛
   - استخدم واجهة برمجة التطبيقات التابعة لعرض البيانات المعالجة/المجمعة لأنظمة SCADA القديمة أو أنظمة الطرف الثالث عبر Modbus.

---

## 5. استخدام واجهتي برمجة التطبيقات الرئيسية والتابعة معًا

نظرًا لأن واجهتي برمجة التطبيقات الرئيسية والتابعة تشتركان في نفس تصميم Builder + Hook، يمكنك بسهولة دمجهما داخل عملية واحدة:

1. استخدم واجهة برمجة التطبيقات الرئيسية لاستقصاء عدة أجهزة أعلى وبناء نموذج بيانات داخلي موحد؛
2. استخدم واجهة برمجة التطبيقات التابعة لتعيين نموذج البيانات ذلك إلى فضاء سجلات Modbus؛
3. دع الأنظمة الخارجية تعامل عمليتك كجهاز Modbus قياسي.

هذا النمط مفيد لبناء بوابات البروتوكولات أو عقد التجميع أو أدوات الاختبار.

---

## 6. تشغيل مثال الجهاز التابع

من جذر المستودع:

```bash
cargo run --package api_slave -- /tmp/vcom2
```

يمكنك إقران هذا مع مثال الجهاز الرئيسي أو Aoba CLI/TUI للاختبار:

- ابدأ مثال الجهاز التابع مستمعًا على `/tmp/vcom2`؛
- ثم استخدم مثال الجهاز الرئيسي أو CLI/TUI لاستقصاء ذلك المنفذ والتحقق من سلوك القراءة/الكتابة.
