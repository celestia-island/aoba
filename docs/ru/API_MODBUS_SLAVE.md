# Руководство по использованию API Modbus Slave

Этот документ описывает использование API Modbus Slave библиотеки Aoba из Rust-приложений для предоставления данных устройствам Modbus master. Типичные сценарии включают промышленные производственные линии, системы управления технологическими процессами и испытательные стенды.

Справочный пример — крейт `examples/api_slave`.

## 1. Обзор

Aoba предоставляет API стороны ведомого устройства (slave), аналогичный по стилю API master, основанный на шаблоне Builder + Hook. Он полезен, когда вам нужно:

- Превратить ваш процесс в ведомое устройство Modbus, предоставляя данные coils/регистров внешним master-устройствам;
- Быстро создать настраиваемое устройство Modbus для интеграционных тестов или моделирования;
- Подключить цепочку промежуточных перехватчиков для логирования, статистики, контроля доступа и оповещений.

Основная точка входа — `_main::api::modbus::ModbusBuilder`, но используются методы `new_slave` / `build_slave`:

```rust
use _main::api::modbus::{ModbusBuilder, ModbusHook, ModbusResponse, RegisterMode};
```

## 2. Базовый жизненный цикл slave

Упрощённая версия примера slave выглядит следующим образом:

```rust
use anyhow::Result;
use std::sync::Arc;
use _main::api::modbus::{ModbusBuilder, ModbusHook, ModbusResponse, RegisterMode};

struct ResponseLoggingHook;

impl ModbusHook for ResponseLoggingHook {
    fn on_before_request(&self, _port: &str) -> Result<()> {
        Ok(())
    }

    fn on_after_response(&self, port: &str, response: &ModbusResponse) -> Result<()> {
        log::info!(
            "sent response on {}: station={}, addr=0x{:04X}, values={:04X?}",
            port,
            response.station_id,
            response.register_address,
            response.values
        );
        Ok(())
    }

    fn on_error(&self, _port: &str, error: &anyhow::Error) {
        log::warn!("error: {}", error);
    }
}

#[tokio::main]
async fn main() -> Result<()> {
    env_logger::builder()
        .filter_level(log::LevelFilter::Info)
        .init();

    let args: Vec<String> = std::env::args().collect();
    let port = if args.len() > 1 { &args[1] } else { "/tmp/vcom2" };

    let hook: Arc<dyn ModbusHook> = Arc::new(ResponseLoggingHook);

    let _slave = ModbusBuilder::new_slave(1)
        .with_port(port)
        .with_register(RegisterMode::Holding, 0, 5)
        .with_timeout(1000)
        .add_hook(hook)
        .build_slave()?;

    // Держим slave запущенным и ожидающим запросы от master
    tokio::signal::ctrl_c().await?;
    Ok(())
}
```

### Основные параметры конфигурации

- **Порт**: тот же формат, что и для master (`/dev/ttyUSB*`, `/dev/ttyS*`, `/tmp/vcom2` и т.д.);
- **Station ID**: должен соответствовать идентификатору станции, который master-устройства будут использовать при обращении к данному slave;
- **Режим регистров и диапазон адресов**: определяют, какая часть адресного пространства Modbus предоставляется данным slave;
- **Таймаут**: используется внутри для управления таймаутами ввода-вывода/обработки (обычно согласуется с настройками master).

## 3. Цепочка перехватчиков

На стороне slave также можно зарегистрировать несколько перехватчиков, образующих цепочку middleware. Типичные обязанности:

- Проверка или инспекция входящих запросов до их обработки;
- Логирование и постобработка ответов после их отправки;
- Генерация оповещений или обновление статистики при возникновении ошибок.

Крейт `examples/api_slave` демонстрирует три связанных перехватчика:

- `RequestMonitorHook`: отслеживает запросы и логирует/оповещает об ошибках;
- `ResponseLoggingHook`: логирует каждый ответ с адресом регистра и значениями;
- `StatisticsHook`: отслеживает количество запросов.

Этот шаблон позволяет вынести сквозные задачи (логирование, метрики, контроль доступа, ограничение частоты запросов и т.д.) из основной бизнес-логики и подключать их декларативно к экземпляру slave.

## 4. Типичные сценарии использования

Типичные сценарии использования API slave в промышленных условиях и тестовых стендах:

1. **Программный симулятор устройств**
   - Когда реальные устройства ещё недоступны, смоделируйте устройство Modbus на Rust;
   - Периодически обновляйте внутренние значения регистров в соответствии с тестовыми сценариями;
   - Выполняйте сквозные интеграционные тесты в CI.
2. **Слой адаптации протокола**
   - Ваши реальные устройства могут использовать CAN, проприетарный TCP или другую полевую шину, тогда как вышестоящие системы ожидают Modbus;
   - Используйте API slave для отображения этих сигналов в пространство регистров/coils Modbus и предоставления единого интерфейса Modbus.
3. **Пограничный шлюз для предоставления обработанных данных**
   - Собирайте и нормализуйте данные из нескольких источников внутри вашего процесса или шлюза;
   - Используйте API slave для предоставления обработанных/агрегированных данных устаревшим SCADA-системам или сторонним системам через Modbus.

---

## 5. Совместное использование API master и slave

Поскольку API master и slave используют один и тот же шаблон проектирования Builder + Hook, их легко комбинировать в рамках одного процесса:

1. Используйте API master для опроса нескольких вышестоящих устройств и построения единой внутренней модели данных;
2. Используйте API slave для отображения этой модели данных в адресное пространство регистров Modbus;
3. Позвольте внешним системам рассматривать ваш процесс как стандартное устройство Modbus.

Этот шаблон полезен для создания шлюзов протоколов, узлов агрегации или тестовых обвязок.

## 6. Запуск примера slave

Из корня репозитория:

```bash
cargo run --package api_slave -- /tmp/vcom2
```

Это можно использовать совместно с примером master или Aoba CLI/TUI для тестирования:

- Запустите пример slave, ожидающий подключений на `/tmp/vcom2`;
- Затем используйте пример master или CLI/TUI для опроса этого порта и проверки поведения чтения/записи.
