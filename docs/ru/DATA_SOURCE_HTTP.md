# Пользовательский источник данных — HTTP

## Быстрый старт — запуск небольшого CLI-приёмника

Запустите CLI приложения в режиме приёмника (CLI создаст HTTP-эндпоинт и будет обрабатывать отправленные JSON-данные). Пример (запуск из корня репозитория):

```bash
# с использованием cargo (рекомендуется при разработке)
cargo run --bin aoba -- --master-provide-persist /tmp/vcom1 \
  --register-mode holding --register-address 0 --register-length 10 \
  --data-source http://8080

# или, если бинарный файл собран:
./target/debug/aoba --master-provide-persist /tmp/vcom1 --data-source http://8080
```

Приведённая выше команда запускает HTTP-сервер, привязанный к `127.0.0.1:8080`, принимающий `POST`-запросы по адресу `/` (корень). Используйте пример с `curl` ниже для отправки данных.

## Обзор

Этот документ описывает пользовательский источник данных HTTP, используемый приложением. Здесь представлен ожидаемый формат запроса, стандартные заголовки и простой пример с `curl` для быстрой проверки интеграции.

## Эндпоинт

- Метод: `POST`
- URL: `http://<хост>:<порт>/` (пример: `http://localhost:8080/`)
- Content-Type: `application/json`

## Формат запроса

Сервис принимает тело в формате JSON. Минимальный пример полезной нагрузки:

```json
{
  "source": "http",
  "timestamp": "2025-11-15T12:34:56Z",
  "port": "/tmp/vcom1",
  "payload": {
    "type": "register_update",
    "registers": [
      {"address": 0, "value": "1234"},
      {"address": 1, "value": "abcd"}
    ]
  }
}
```

Примечания:

- Используйте формат ISO 8601 для `timestamp`, когда это возможно.
- Содержимое `payload` зависит от конкретного приложения; приведённый выше пример показывает типичное обновление в стиле регистров.

## Пример теста с curl

Замените `<хост>` и `<порт>` на данные вашего запущенного сервера. Эта команда `curl` отправляет приведённую выше JSON-нагрузку:

```bash
curl -v -X POST "http://localhost:8080/" \
  -H "Content-Type: application/json" \
  -d '{
    "source":"http",
    "timestamp":"2025-11-15T12:34:56Z",
    "port":"/tmp/vcom1",
    "payload":{
      "type":"register_update",
      "registers":[{"address":0,"value":"1234"}]
    }
  }'
```

## Ожидаемое поведение

- HTTP `200 OK` (или `202 Accepted`) для принятых/поставленных в очередь сообщений.
- Если сервер возвращает ошибку (4xx/5xx), проверьте тело ответа для получения подробностей.

## Советы и устранение неполадок

- Убедитесь, что заголовок `Content-Type: application/json` присутствует.
- Если ваш сервер требует аутентификацию, добавьте соответствующий заголовок `Authorization` (например, `Bearer <токен>`).
- Для больших полезных нагрузок рекомендуется использовать `--data-binary` и увеличить таймауты сервера.

Если вам нужен специализированный пример, соответствующий внутренней схеме, предоставьте пример JSON, и разработчики адаптируют обработчик эндпоинта соответствующим образом.
