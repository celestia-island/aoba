# Пользовательский источник данных — MQTT

## Быстрый старт — запуск небольшого CLI-приёмника

Запустите CLI приложения так, чтобы он подписался на топик MQTT и работал в качестве приёмника. Пример (запуск из корня репозитория):

```bash
# с использованием cargo (рекомендуется при разработке)
cargo run --bin aoba -- --master-provide-persist /tmp/vcom1 \
  --register-mode holding --register-address 0 --register-length 10 \
  --data-source mqtt://localhost:1883/aoba/data/in

# или, если бинарный файл собран:
./target/debug/aoba --master-provide-persist /tmp/vcom1 --data-source mqtt://localhost:1883/aoba/data/in
```

URL `mqtt://.../<топик>` содержит путь к топику (например, `aoba/data/in`), и CLI подпишется на этот топик.

## Обзор

Этот документ описывает публикацию сообщений в пользовательский источник данных приложения на основе MQTT. Включает конфигурацию брокера/подключения, рекомендуемые имена топиков и пример полезной нагрузки `mosquitto_pub` для передачи нисходящих данных.

## Брокер / подключение

- Хост: `mqtt.example.com` или `localhost`
- Порт: `1883` (без шифрования) или `8883` (TLS)
- Имя пользователя/пароль: опционально — если ваш брокер требует аутентификацию, укажите их в конфигурации клиента
- TLS: при использовании `8883` предоставьте CA-сертификат и клиентский сертификат/ключ при необходимости

## Рекомендуемые топики

- Входящий (к приложению): `aoba/data/in` — приложение подписывается на этот топик для получения восходящих данных или команд
- Нисходящий (к устройству/vcom): `aoba/data/out/<порт>` — приложение публикует обработанные нисходящие сообщения, адресованные конкретному порту (например, `aoba/data/out/tmp_vcom1`)

## Формат полезной нагрузки

Приложение ожидает полезные нагрузки в формате JSON. Точная схема гибкая, но следующий пример представляет практичный формат для обновлений статуса и нисходящих команд:

```json
{
  "source": "mqtt",
  "timestamp": "2025-11-15T12:34:56Z",
  "port": "/tmp/vcom1",
  "type": "downlink",
  "body": {
    "command": "write_register",
    "registers": [{"address":0, "value": "1234"}]
  }
}
```

## Пример: публикация нисходящих данных с помощью mosquitto_pub

Этот пример публикует нисходящее сообщение во входящий топик, которое приложение обработает и выполнит физическую запись на настроенный `порт`.

```bash
mosquitto_pub -h localhost -p 1883 -t "aoba/data/in" -u "user" -P "pass" -m '{
  "source":"mqtt",
  "timestamp":"2025-11-15T12:34:56Z",
  "port":"/tmp/vcom1",
  "type":"downlink",
  "body":{ "command":"write_register", "registers":[{"address":0,"value":"1234"}] }
}'
```

## Заметки и советы

- Используйте предсказуемые имена топиков для упрощения фильтрации и управления правами.
- При указании пути к физическому последовательному порту (например, `/tmp/vcom1`) избегайте символов, которые могут вызвать проблемы при разборе топика; можно сопоставить имена портов с безопасными для топиков метками в конфигурации.
- Если ваш брокер поддерживает удержанные сообщения (retained), будьте осторожны: удержанные нисходящие сообщения могут быть повторно применены при повторном подключении.

При необходимости можно предоставить пример конфигурации брокера или автоматизированную тестовую обвязку (например, скрипт, публикующий последовательность нисходящих сообщений и ожидающий подтверждения статуса от CLI/TUI).
