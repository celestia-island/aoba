# Руководство по использованию API Modbus Master

Этот документ описывает использование API Modbus Master библиотеки Aoba из Rust-приложений в типичных промышленных сценариях (мониторинг производственных линий, управление технологическими процессами, контроль параметров окружающей среды и т.д.). В качестве справочного примера используется крейт `examples/api_master`.

## 1. Обзор

Aoba предоставляет API Modbus master на основе типажей, предназначенный для встраивания в другие Rust-приложения или программное обеспечение управления оборудованием. Типичные сценарии использования:

- Периодический опрос ведомых устройств Modbus (RTU через последовательные или виртуальные порты)
- Сбор значений coils / регистров в собственную систему телеметрии или логику управления
- Интеграция с существующими системами логирования / мониторинга через перехватчики (hooks)

Основная точка входа — тип `ModbusBuilder` из `_main::api::modbus`.

```rust
use _main::api::modbus::{ModbusBuilder, ModbusHook, ModbusResponse, RegisterMode};
```

> Примечание: в примерах корень крейта называется `_main`. В вашем проекте это обычно основной крейт `aoba` или имя, указанное в `Cargo.toml`.

## 2. Базовый жизненный цикл master

Минимальный цикл опроса master выглядит следующим образом:

```rust
use anyhow::Result;
use std::time::Duration;
use _main::api::modbus::{ModbusBuilder, RegisterMode};

fn main() -> Result<()> {
    let master = ModbusBuilder::new_master(1) // station id of the slave
        .with_port("/dev/ttyUSB0")          // или `/tmp/vcom1` и т.д.
        .with_register(RegisterMode::Holding, 0, 10)
        .with_timeout(1000)                  // миллисекунды
        .build_master()?;

    loop {
        if let Some(resp) = master.recv_timeout(Duration::from_secs(1)) {
            println!("values = {:04X?}", resp.values);
        }
    }
}
```

### Важные параметры

- **Порт**: любой последовательный или виртуальный порт, который Aoba может открыть (реальный `/dev/ttyUSB*`, `/dev/ttyS*` или виртуальный `/tmp/vcom*`, созданный через socat).
- **Station ID**: адрес ведомого устройства Modbus (обычно 1–247).
- **Режим регистров**: один из `RegisterMode::Coils`, `DiscreteInputs`, `Holding`, `Input`.
- **Адрес / длина регистров**: начальный адрес и количество считываемых элементов, соответствующие таблице адресов Modbus вашего устройства (например, ПЛК или шлюза датчиков).
- **Таймаут**: таймаут запроса в миллисекундах.

Master внутренне запускает цикл опроса и передаёт ответы в канал; ваш код просто вызывает `recv_timeout` для получения новых данных.

## 3. Использование перехватчиков для логирования и мониторинга

Для промышленных систем (производственные линии, технологическое оборудование, локальные датчики и т.д.) обычно требуется:

- Логировать каждый успешный ответ
- Отслеживать ошибки и таймауты
- Возможность отправки данных в шину сообщений или базу данных

Типаж `ModbusHook` позволяет подключить эту логику централизованно.

```rust
use anyhow::Result;
use std::sync::Arc;
use _main::api::modbus::{ModbusBuilder, ModbusHook, ModbusResponse, RegisterMode};

struct LoggingHook;

impl ModbusHook for LoggingHook {
    fn on_before_request(&self, port: &str) -> Result<()> {
        log::debug!("sending request on {}", port);
        Ok(())
    }

    fn on_after_response(&self, port: &str, resp: &ModbusResponse) -> Result<()> {
        log::info!(
            "resp {}: station={}, addr=0x{:04X}, values={:04X?}",
            port,
            resp.station_id,
            resp.register_address,
            resp.values,
        );
        Ok(())
    }

    fn on_error(&self, port: &str, err: &anyhow::Error) {
        log::warn!("modbus error on {}: {}", port, err);
    }
}

fn main() -> Result<()> {
    env_logger::init();

    let master = ModbusBuilder::new_master(1)
        .with_port("/tmp/vcom1")
        .with_register(RegisterMode::Holding, 0, 5)
        .with_timeout(1000)
        .add_hook(Arc::new(LoggingHook))
        .build_master()?;

    // теперь опрос через recv_timeout, как в базовом примере
    # let _ = master;
    Ok(())
}
```

Вы можете зарегистрировать несколько перехватчиков (например, один для логирования, другой для экспорта метрик).

## 4. Шаблон интеграции для промышленного мониторинга оборудования

Для типичных сценариев промышленного мониторинга (производственные линии, технологические установки, устройства контроля окружающей среды и т.д.) рекомендуется следующий шаблон:

1. **Настройте порты и станции** через TUI или CLI Aoba, либо задайте их программно в приложении.
2. **Создайте по одному master на каждый физический/виртуальный порт** с помощью `ModbusBuilder::new_master`.
3. **Запустите по задаче Tokio на каждый master**, которая:
   - вызывает `recv_timeout` в цикле
   - преобразует `ModbusResponse::values` в инженерные величины (давление, температура, состояние клапанов и т.д.)
   - передаёт обработанные данные в систему мониторинга (MQTT, HTTP, база данных и т.д.)
4. Используйте `ModbusHook` для централизации логирования, измерения задержки и подсчёта ошибок.

Поскольку Aoba построен на `tokio`, API master предназначен для использования внутри асинхронной среды выполнения, но предоставляет удобный метод `recv_timeout` с блокирующим стилем вызова.

## 5. Обработка ошибок и таймаутов

- `build_master()` возвращает `anyhow::Error`, если порт не удалось открыть или конфигурация недействительна.
- `recv_timeout()` возвращает `None` при таймауте; само по себе это не является ошибкой.
- Ошибки уровня протокола (CRC, коды исключений, ошибки ввода-вывода) сообщаются через `ModbusHook::on_error`.

Рекомендуемый подход:

- Рассматривайте случайные таймауты как нормальное явление в нестабильных последовательных каналах связи.
- Используйте счётчик в перехватчике; при превышении порога последовательных ошибок генерируйте аварийное оповещение.

## 6. Запуск примера

Из корня репозитория:

```bash
cargo run --package api_master -- /tmp/vcom1
```

 В условиях, приближённых к промышленным (например, стенд хранения водорода), обычно:

- Используют Aoba CLI/TUI или `examples/modbus_slave` для имитации стороны ведомого устройства.
- Затем запускают пример `api_master` для проверки корректности подключения Modbus и логики на уровне приложения.

## 7. Master в ручном режиме (poll_once / операции записи)

Для сценариев, где требуется точный контроль времени опроса (конечные автоматы, адаптивные стратегии или операции записи), используйте `build_master_manual()`:

```rust
use anyhow::Result;
use _main::api::modbus::{ModbusBuilder, RegisterMode};

fn main() -> Result<()> {
    let master = ModbusBuilder::new_master(1)
        .with_port("/dev/ttyUSB0")
        .with_baud_rate(9600)
        .with_timeout(5000)
        .build_master_manual()?;

    // Разовый опрос вручную
    let response = master.poll_once(RegisterMode::Holding, 0x00, 10)?;
    println!("Values: {:?}", response.values);

    // Запись одного holding-регистра (fc 0x06)
    master.write_holding(0x00, 0x1234)?;

    // Запись нескольких holding-регистров (fc 0x10)
    master.write_registers(0x00, &[0x1234, 0x5678, 0x9ABC])?;

    // Запись coils (fc 0x0F)
    master.write_coils(0x00, &[true, false, true, true])?;

    Ok(())
}
```

### Когда использовать ручной режим

| Сценарий | Рекомендуемый режим |
|----------|---------------------|
| Непрерывный мониторинг / сбор данных | `build_master()` (автоматический) |
| Циклы управления «чтение-модификация-запись» | `build_master_manual()` |
| Конечные автоматы / событийный опрос | `build_master_manual()` |
| Адаптивный опрос на основе задержки ответа | `build_master_manual()` |
| Разовая диагностика или конфигурирование | `build_master_manual()` |

### Детали операций записи

- **`write_holding(address, value)`** — записывает один holding-регистр с использованием кода функции 0x06. Оптимально для записи отдельных параметров конфигурации.
- **`write_registers(address, values)`** — записывает несколько последовательных holding-регистров с использованием кода функции 0x10. Оптимально для пакетной записи параметров.
- **`write_coils(address, values)`** — записывает несколько coils с использованием кода функции 0x0F. Включает автоматическую перестановку байтов для записи 11 coils (требуется некоторым устройствам).
- Все методы записи блокируются до подтверждения от ведомого устройства или возникновения ошибки.
